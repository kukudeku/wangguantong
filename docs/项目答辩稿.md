# 网管通电竞网吧运营管理系统答辩稿

> 建议时长：8～10 分钟。方括号内容为演示提示，不需要朗读。

## 一、开场

尊敬的各位老师，大家好。

我本次答辩的项目是《网管通——电竞网吧运营管理系统》。它是一个基于 Spring Boot 3 和 Vue 3 的前后端分离系统，面向网吧会员、机位、点餐、营销和现场服务等运营场景。

传统网吧大量业务依赖前台人工处理，容易出现机位状态更新不及时、计费不透明、预约冲突、订单遗漏、优惠核销重复以及服务诉求无法追踪等问题。网管通通过用户自助端和管理员后台共享同一套业务数据，把上机、支付、营销和服务工单形成闭环。

## 二、系统目标与架构

系统有三个核心目标：

1. 统一管理会员、电脑、上机、预约、点餐、支付、优惠和服务记录；
2. 让顾客自主完成上机、点餐、签到、团购券核销和呼叫网管；
3. 让管理员通过后台处理基础资料、订单、工单和营销规则。

前端使用 Vue 3、Vue Router、Axios、Vite 5 和 Arco Design Vue；后端使用 Java 17、Spring Boot 3.3.5、MyBatis-Plus 3.5.7 和 MySQL 8。后端按 Controller、Service、Mapper、Entity 分层，统一业务响应为 `code`、`message`、`data`。

当前代码共有 14 个 Controller、64 个 HTTP 接口，完整数据库有 18 张表。本地前端端口是 `5173`，后端端口是 `8087`。

## 三、用户端功能

[演示：打开 `/`，使用测试身份证号登录]

用户端目前有六组功能。

第一组是上机服务，包括机位大厅、当前上机、我的预约和上机记录。座位图按区域展示空闲、使用中、预约锁定、维修中四种状态。只有空闲电脑可以上机或预约。

第二组是点餐服务。用户可按分类选择商品，购物车批量结算，选择优惠券，并使用余额、微信 Native 扫码或支付宝电脑网站支付。微信和支付宝默认关闭，只有配置真实商户资料后才启用。

第三组是会员活动，包括每日签到和我的优惠券。管理员可以配置连续签到第几天发放哪张优惠券，用户达到条件后自动领取。

第四组是推广计划。每个会员有唯一邀请码，新用户注册时填写有效邀请码，系统按后台规则分别奖励邀请人和新用户，并记录邀请明细。

第五组是服务中心。用户可以呼叫网管、提交故障报修并查看处理记录。故障报修与电脑维修状态联动。

第六组是账户服务，包括余额明细、微信/支付宝余额充值、团购券核销、修改密码和退出登录。用户充值先生成第三方支付单，只有平台验签回调或主动查单确认成功后才增加余额；同一支付单只到账一次。团购券必须由后台预先录入，每个券码只能成功核销一次。

## 四、管理员后台功能

[演示：访问 `/admin`]

管理员后台按业务分成七组：数据概览、会员服务、机位管理、服务中心、上机运营、点餐管理和营销活动。

首页看板显示会员总数、电脑总数、空闲和使用中电脑、今日充值金额、今日上机次数。会员模块支持资料维护和余额充值；机位与上机模块维护电脑状态、预约和进行中记录。

点餐后台既能维护商品，也能查看订单的优惠前金额、优惠金额、实付金额、支付方式和支付状态。管理员完成订单表示已交付；取消余额订单会退余额，取消第三方已支付订单会发起原路退款。

营销后台管理团购券、优惠券模板、签到奖励规则和推广奖励金额。服务工单后台可以受理、完成或取消呼叫网管和故障报修。

## 五、重点业务实现

### 1. 实时计费与自动下机

开始上机时，后端校验会员状态、重复上机、电脑状态和首小时余额，然后立即扣除首小时费用。计费公式为：

```text
累计费用 = 每小时单价 × 向上取整的计费小时 × 会员折扣
```

普通、黄金、钻石会员折扣分别为 1、0.9、0.8。后端每 60 秒重新计算累计费用，只扣“当前应收减去已扣金额”的差额，避免重复扣款。余额不足时自动结束上机记录并释放电脑。

### 2. 预约状态闭环

用户只能预约当前至未来一小时内的空闲电脑，同一用户只能有一条有效预约。预约后电脑改为预约锁定；取消或超时后恢复空闲；预约上机时再次校验归属和有效期，并复用普通上机计费逻辑。

### 3. 优惠券与三种支付

一次购物车结算使用同一个批次号。后端重新读取商品价格，计算优惠前金额，校验优惠券门槛，并把优惠金额分摊到订单明细。

余额支付立即扣款；点餐和余额充值的微信、支付宝订单先保存为待支付，并通过 `business_type` 区分业务。平台异步回调验签成功或主动查单确认成功后，点餐订单才变为已下单，充值金额才写入余额。默认 15 分钟未支付会自动关闭；点餐同时归还优惠券。退款和充值到账都通过独立业务单号防止重复处理。

### 4. 营销防重复

签到按会员和日期唯一，一天只能签到一次；优惠券使用通过状态条件更新防止重复提交；团购券核销同样只允许从未使用状态更新为已使用；邀请记录对新用户设置唯一约束，保证每个新用户只奖励一次。

### 5. 服务工单与机位联动

呼叫网管可以只填写所在位置；故障报修必须选择电脑。同一电脑不能同时存在多条未完成报修。管理员受理故障后电脑进入维修中，完成或取消后恢复空闲；正在使用或预约锁定的电脑必须先释放。

## 六、数据库与事务

18 张表按业务分为账号机位、上机预约、点餐支付、优惠券签到、推广邀请、团购券充值和服务工单。数据库没有声明物理外键，关系由 Service 层校验和事务维护。

充值、上机、预约、点餐、优惠券、团购券、签到、邀请奖励和退款等跨表操作使用事务。金额统一使用 `BigDecimal` 和 `DECIMAL`，避免浮点误差。订单和业务记录保留会员姓名、电脑编号、商品名称及金额快照，防止基础资料修改影响历史记录。

## 七、测试与部署

后端可执行 `mvn test`，当前包含会员邀请码生成和支付服务相关测试；前端可执行 `npm run build` 生成生产静态文件。

本地由 Vite 将 `/api` 代理到 `8087`。宝塔部署时，Nginx 提供前端静态文件，并把 `/api` 代理到 `127.0.0.1:8087`。真实支付需要公网 HTTPS 回调、商户资料和放在网站目录之外的密钥文件。

## 八、当前不足

第一，密码仍是明文保存，登录状态主要依赖 `localStorage`，后端没有统一鉴权。生产化应加入 BCrypt、Spring Security、服务端会话或 JWT 和接口授权。

第二，当前列表不分页，适合课堂数据量；数据规模扩大后需要分页、索引和缓存。

第三，并发抢机位、重复预约等场景主要依赖业务层状态检查，生产环境应增加条件更新、乐观锁或唯一约束。

第四，自动化测试目前集中在少数后端服务，后续应补充 Controller 集成测试和浏览器端到端测试。

第五，真实支付代码已经接入，但最终联调仍依赖真实商户资质、证书、网关和 HTTPS 环境。

# 常见答辩问题

## 1. 为什么使用前后端分离？

页面交互与业务规则可以独立维护，用户端和后台复用同一套接口，以后增加其他终端也可复用后端。

## 2. 实时扣费如何避免重复？

`online_record.total_amount` 保存累计已扣金额，每次只扣当前累计应收与已扣金额的差额。

## 3. 为什么第三方支付不能在前端点击后直接算成功？

前端结果可能伪造或中断，必须由后端验证支付平台异步通知签名或主动查询结果后确认。

## 4. 优惠券怎样避免重复使用？

后端通过“编号匹配且状态仍为可使用”的条件更新占用优惠券，更新失败说明已被其他请求使用。

## 5. 为什么数据库不使用外键？

课程项目为简化迁移，当前使用逻辑关联和事务；生产环境可以根据删除策略和数据规模补充外键、索引或归档策略。

## 6. 当前最大的安全问题是什么？

密码未散列且后端没有统一身份认证。正式部署前必须优先修复鉴权、密码、授权和审计。

## 7. 为什么说真实支付“已支持”但默认不可用？

代码已经实现支付单、二维码/收银台、回调验签、查询、超时关闭和退款；但商户资质、密钥、证书和 HTTPS 属于部署环境，未配置时平台保持关闭。

## 8. 64 个接口如何统计？

按 14 个 Controller 中实际存在的 `GET/POST/PUT/DELETE` 映射统计，不包含前端路由和 Service 方法；新增项为 `POST /recharge/payment`。

## 9. 为什么取消一件商品只退该明细金额？

购物车可能包含多条商品且共用一张优惠券，系统把优惠分摊到明细，退款时只退该明细实付金额，避免超退。

## 10. 服务工单为何与电脑状态联动？

故障报修受理后机位不能继续分配，因此进入维修中；完成或取消后释放，保证工单状态与可用机位一致。
