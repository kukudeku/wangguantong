# 网管通电竞网吧运营管理系统答辩稿

> 建议答辩时长：8～10 分钟  
> 使用方式：正文可以直接照读；方括号中的内容是演示提示，不需要念出来。

## 一、开场

尊敬的各位老师，大家好。

我本次答辩的项目是《网管通——电竞网吧运营管理系统》。这是一个基于 Spring Boot 和 Vue 3 开发的前后端分离项目，主要面向中小型网吧的日常运营场景。

传统网吧的会员充值、电脑分配、上机计费、座位预约和商品点餐，往往需要由前台人工完成。当业务集中时，容易出现机位状态更新不及时、计费过程不透明、订单遗漏以及用户等待时间较长等问题。因此，我设计了网管通系统，把管理端和用户自助端结合起来，让管理员能够统一维护业务数据，也让用户能够自行完成上机、预约、点餐、充值和记录查询。

下面我将从项目目标、系统架构、核心功能、重点业务实现、数据库设计、测试部署以及项目不足七个方面进行介绍。

## 二、项目目标

本项目主要有三个目标。

第一，实现网吧基础资料的数字化管理，包括会员、电脑机位、商品以及各类业务记录。

第二，实现完整的上机业务闭环。用户选择空闲电脑后开始计费，系统实时扣除余额，余额不足时自动下机，并释放电脑。

第三，提供用户自助服务。用户使用身份证号和密码登录后，可以查看座位图、自助上机、预约电脑、自助点餐、模拟充值，并查看自己的余额明细、上机记录和订单记录。

系统默认入口是用户端，访问 `/user`；管理员后台入口是 `/admin`。两个系统使用独立的登录状态和页面布局，避免普通用户进入后台管理页面。

## 三、系统架构与技术选型

项目采用前后端分离架构。

前端使用 Vue 3、Vue Router、Axios、Vite 和 Arco Design Vue。Vue 3 负责页面和响应式数据，Vue Router 负责用户端与后台页面跳转，Axios 负责调用后端接口，Arco Design 提供表单、表格、弹窗和消息提示等组件，Vite 用于本地开发和生产构建。

后端使用 Java 17、Spring Boot 3、MyBatis-Plus 和 MySQL。Spring Boot 提供 REST 接口和业务运行环境，MyBatis-Plus 简化常见的增删改查，MySQL 负责保存会员、电脑、充值、上机、预约和订单数据。

后端代码按照 Controller、Service、Mapper 和 Entity 分层。

Controller 层负责接收前端请求并返回统一结果；Service 层负责余额校验、状态流转和实时计费等业务规则；Mapper 层负责访问数据库；Entity 层负责 Java 对象与数据库表的映射。

所有接口统一返回 `code`、`message` 和 `data`。前端通过 Axios 响应拦截器统一判断请求是否成功，失败时直接显示后端返回的提示，减少了每个页面重复编写错误处理代码。

## 四、系统主要功能

### 1. 用户端功能

[演示：打开用户登录页面]

用户使用身份证号作为登录账号。系统同时支持用户注册和登录后的密码修改。登录成功后，用户端分为座位上机、自助点餐和消费记录三个主要模块。

[演示：进入座位上机页面]

座位图按照电脑区域分组，并使用不同颜色显示空闲、使用中、预约锁定和维修中四种状态。只有空闲电脑可以上机或预约。用户点击上机后，需要进行二次确认，确认成功才会开始计费。

如果用户已经上机，页面不再显示其他电脑，而是只显示当前电脑编号、上机时长、计费小时、当前消费和账户余额，并提供自助下机按钮。会员状态旁边也会显示“上机中”，让当前状态更加明确。

[演示：进入自助点餐页面]

商品按照饮料、餐食、零食和其他分类展示。每个商品默认数量为零，用户通过加号和减号选择数量。选中的商品会进入购物车，购物车显示商品清单、数量、小计和总金额，用户最后统一点击“下单”。会员余额不足时，系统会阻止下单并提示充值。

[演示：进入消费记录页面]

消费记录包括余额明细、上机记录和订单记录。用户可以查看每次充值和消费的时间、金额及说明，也可以查看历史上机电脑、扣费金额和订单状态。

用户端还提供模拟充值功能。由于这是课堂项目，没有接入微信或支付宝，用户选择充值金额并确认后，系统直接增加账户余额，同时生成充值明细。

### 2. 管理员后台功能

[演示：访问 `/admin` 并登录]

后台首页显示会员总数、电脑总数、空闲电脑、使用中电脑、今日充值金额和今日上机次数，并提供电脑座位图，帮助管理员快速了解当前运营状态。

会员管理支持按姓名、身份证号和手机号查询，也支持新增、编辑、充值和删除。管理员新增用户时必须设置密码。会员分为普通会员、黄金会员和钻石会员，其中普通会员不打折，黄金会员上机九折，钻石会员上机八折。散客充值后会自动转为普通会员。

电脑管理支持维护电脑编号、所在区域、每小时单价、状态和备注。预约、上机和下机会自动改变电脑状态，管理员不需要手工重复操作。

上机管理只显示当前正在上机的记录，历史记录放在“上机历史”标签中。管理员可以查看实时上机时长、当前消费、余额提醒，也可以执行下机结算。

预约管理支持创建、取消和预约上机。点餐管理分为订单管理和商品维护，管理员可以设置商品名称、分类、价格和上下架状态，并对已下单订单执行“完成订单”或“取消订单”。会员订单被取消时，系统会退回原订单金额。

## 五、核心业务实现

### 1. 实时计费与自动下机

本项目最核心的业务是实时上机计费。

用户开始上机时，后端首先检查会员是否存在、账号状态是否正常、是否已经有其他进行中的上机记录，以及目标电脑是否空闲。全部通过后，系统根据电脑每小时单价和会员等级计算首小时费用。

计费公式是：

`累计费用 = 每小时单价 × 计费小时 × 会员折扣`

计费小时采用向上取整，不足一小时按一小时计算。例如每小时 6 元，普通会员上机 61 分钟，计费小时为 2 小时，累计费用为 12 元。

系统不会等到下机后一次性扣费。开始上机时立即扣除首小时费用，之后通过 Spring 的定时任务每 60 秒检查进行中的上机记录。系统用当前累计应收金额减去已经扣除的金额，只补扣新增差额，因此不会重复扣费。

如果余额不足以支付新的计费小时，系统会自动把上机记录改为已完成，记录结束时间，并把电脑状态恢复为空闲，从而避免继续产生无法支付的费用。

### 2. 预约锁定与超时取消

用户只能预约当前时间起一小时以内的电脑。预约成功后，会新增一条“已预约”记录，同时把电脑改为“预约锁定”，其他用户无法操作该机位。

系统每 60 秒扫描一次已超过预约时间但仍未上机的记录，并自动取消预约、释放电脑。用户点击预约上机时，后端还会再次检查预约是否超时，然后复用普通上机流程完成余额校验和首小时扣费。这样可以避免预约上机和普通上机使用两套不同的计费规则。

### 3. 事务与数据一致性

充值、上机、预约和会员点餐都会同时修改多项数据。例如充值需要增加会员余额并新增充值记录；上机需要扣除余额、新增上机记录并修改电脑状态。

这些核心操作使用 Spring 事务处理。如果其中任意一步发生异常，前面的数据库修改会一起回滚，避免出现“余额已经扣除但上机记录没有生成”或“预约已经保存但电脑没有锁定”等数据不一致问题。

### 4. 页面实时刷新

用户上机后，前端每秒更新本地显示时长，每 30 秒向后端同步最新余额、消费金额和上机状态。每秒显示时间不需要请求服务器，可以减少网络压力；涉及真实余额的数据则以后端结果为准。

## 六、数据库设计

系统共设计了 8 张主要数据表。

`admin` 表保存管理员账号；`member` 表保存用户登录资料、会员等级、状态和余额；`computer` 表保存机位区域、价格和状态；`recharge_record` 表保存充值明细；`online_record` 表保存上下机时间和已扣金额；`reservation_record` 表保存预约时间和预约状态；`food_item` 表保存商品资料；`food_order` 表保存点餐订单。

在订单表和业务记录表中，我保留了会员姓名、电脑编号、商品名称和下单时价格等快照字段。这样即使以后修改会员资料、电脑编号或商品价格，历史记录仍然能够保持当时的真实信息。

对于金额字段，后端使用 `BigDecimal`，数据库使用 `DECIMAL`，避免使用浮点数计算人民币时产生精度误差。

## 七、测试与部署

在功能测试方面，我按照完整业务场景进行验证，包括用户注册登录、正常会员上机、余额不足禁止上机、实时扣费、余额不足自动下机、预约超时取消、会员折扣、散客充值转会员、点餐扣款、订单取消退款以及记录查询。

在构建方面，后端可以通过 Maven 打包为 JAR 文件，前端可以通过 Vite 生成 `dist` 静态文件。项目既可以在本地运行，也提供了宝塔部署文档。生产部署时，前端由 Nginx 提供静态访问，并通过 `/api` 反向代理到 Spring Boot 后端。

## 八、项目不足与改进方向

本项目目前定位为课堂作业版系统，重点是完整业务流程和前后端协作，因此仍有以下改进空间。

第一，目前密码是简单保存和比较，登录状态保存在浏览器本地。正式系统应使用 BCrypt 加密密码，并使用 Spring Security 配合 Session 或 JWT 完成后端身份认证和接口权限控制。

第二，充值功能目前是模拟充值，没有接入真实支付平台。后续可以增加支付订单、异步回调、签名验证和重复回调处理。

第三，目前主要采用手工业务测试。后续可以为计费、退款和预约超时增加 JUnit 单元测试，并使用接口测试和端到端测试覆盖完整流程。

第四，在多人同时抢同一台电脑的高并发场景下，可以增加数据库乐观锁、唯一约束或分布式锁，进一步避免重复上机和重复预约。

第五，可以继续增加库存管理、交接班、营业报表、优惠券和消息提醒等功能，使系统更接近真实网吧运营环境。

# 现场演示建议顺序

答辩时不要把所有页面逐个点一遍，建议控制在 3～4 分钟：

1. 打开用户登录页，说明身份证号登录与注册功能。
2. 登录用户端，展示账户摘要和不同状态的座位图。
3. 选择空闲电脑，展示二次确认和上机后的当前电脑页面。
4. 展示实时上机时长、当前消费、余额以及下机按钮。
5. 打开自助点餐，选择两个商品，展示购物车清单并下单。
6. 打开消费记录，展示余额、上机和订单三个标签。
7. 进入 `/admin`，展示首页统计和座位图。
8. 展示会员管理、当前上机/上机历史、订单完成和预约管理。

演示前建议准备：

- 至少一台空闲电脑。
- 一个余额充足的正常会员。
- 两到三个已上架且带分类的商品。
- 一条历史上机记录和一条订单记录。
- 避免使用 SQL 初始数据中状态与业务记录不一致的机位。

# 常见答辩问题与参考回答

## 1. 为什么选择前后端分离？

前后端分离可以让页面展示和业务处理各自独立。前端专注交互和响应式布局，后端统一处理计费、余额和状态规则；以后如果增加手机端，也可以继续复用现有后端接口。

## 2. 为什么使用 MyBatis-Plus？

本项目有较多基础增删改查。MyBatis-Plus 提供 `BaseMapper` 和 `IService`，可以减少重复 SQL，把开发重点放在实时计费、预约和退款等业务逻辑上。

## 3. 为什么金额使用 BigDecimal？

`double` 使用二进制浮点数，计算人民币时可能产生精度误差。`BigDecimal` 可以明确控制小数位数和四舍五入规则，更适合余额和订单金额。

## 4. 实时扣费怎样避免重复扣款？

上机记录的 `totalAmount` 保存已经扣除的累计金额。每次结算先计算当前累计应收金额，再减去 `totalAmount`，只扣两者的差额，并更新 `totalAmount`，因此定时任务重复执行也不会重复扣首小时费用。

## 5. 为什么上机后还要定时刷新前端？

后端定时任务负责真实扣费，前端本地计时只负责平滑显示时长。前端每 30 秒同步一次后端，可以及时获得真实余额、消费和自动下机状态，同时避免每秒发送请求。

## 6. 如何保证充值或上机不会只完成一半？

涉及多张表的核心操作使用 `@Transactional`。发生异常时，数据库修改会整体回滚，保证余额、业务记录和电脑状态一致。

## 7. 普通会员和高级会员如何计算折扣？

普通会员折扣系数为 1，黄金会员为 0.9，钻石会员为 0.8。累计费用等于小时单价乘计费小时再乘折扣系数，最终金额保留两位小数。

## 8. 预约超时怎样处理？

后端每 60 秒查询状态为“已预约”且预约时间早于当前时间的记录，把它们改为“已取消”，并将仍处于预约锁定的电脑释放为空闲。

## 9. 为什么订单中还要保存商品名称和价格？

这些字段是下单时的业务快照。如果只保存商品 ID，管理员以后修改商品名称或价格，历史订单会失去当时的真实信息。

## 10. 当前系统最大的安全问题是什么？

当前是课堂演示版本，密码未加密，登录状态主要依赖 `localStorage`，后端接口也没有完整的权限认证。正式上线时应优先加入 BCrypt、Spring Security、Session 或 JWT，并在后端验证每个接口的访问权限。

## 11. 如果两名用户同时选择同一台电脑怎么办？

当前业务层会再次检查电脑状态，可以处理普通顺序请求。但在真正高并发场景下仍可能出现竞争，后续应增加数据库乐观锁、条件更新或唯一约束，确保同一机位同一时间只能产生一条有效上机或预约记录。

## 12. 为什么没有接真实支付？

真实支付需要商户资质、签名、异步回调和安全验证，不适合作为本课堂项目的重点。本项目用模拟充值完成余额和充值明细的业务闭环，并在改进方向中保留真实支付扩展方案。
