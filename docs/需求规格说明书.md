## 1. 项目概述

### 1.1 建设目标

网管通用于把顾客自助服务与网吧后台运营连接到同一套业务数据中，目标包括：

1. 统一管理会员、机位、上机、预约、点餐、余额和营销数据；
2. 形成“上机—实时计费—余额不足自动下机—释放机位”闭环；
3. 形成“购物车—优惠券—支付—订单交付—取消退款”闭环；
4. 支持签到优惠券、邀请码奖励和团购券核销；
5. 支持呼叫网管、故障报修和后台工单处理；
6. 支持本地开发和宝塔/Nginx 部署。

### 1.2 系统定位与边界

系统定位为课程设计与业务原型，可用于受控环境演示。当前具备真实微信/支付宝支付接入代码，但平台默认关闭，实际收款需要商户资质、HTTPS 回调和服务器密钥。

当前未实现：

- Spring Security、服务端 Session/JWT 和细粒度接口权限；
- 密码散列、短信验证码和联网实名认证；
- 多门店、库存采购、交接班、发票和完整财务对账；
- 电脑客户端远程开关机、锁屏；
- Redis、消息队列、WebSocket 和高并发分布式控制。

## 2. 角色与入口

| 角色 | 入口 | 主要权限 |
|---|---|---|
| 未登录访客 | `/`、`/user/login` | 注册、用户登录 |
| 正常用户 | `/user` | 上机、预约、点餐、营销活动、服务工单、账户查询 |
| 管理员 | `/admin` | 运营管理、基础资料、工单、营销规则和订单处理 |

用户按 `memberLevel` 分为普通会员、黄金会员、钻石会员；上机折扣分别为 100%、90%、80%。`userType` 仍保留“会员/散客”，散客充值后自动转为普通会员。点餐后台可为散客创建订单，但散客不能使用余额支付。

## 3. 功能结构

### 3.1 用户端

```text
用户自助端
├── 上机服务：机位大厅、当前上机、我的预约、上机记录
├── 点餐服务：商品选购、我的订单
├── 会员活动：每日签到、我的优惠券
├── 推广计划：邀请好友、邀请记录
├── 服务中心：呼叫网管、故障报修、服务记录
└── 账户服务：余额账户、团购验券、修改密码、退出
```

### 3.2 管理员后台

```text
管理员后台
├── 数据概览：首页看板
├── 会员服务：会员列表、余额充值
├── 机位管理：电脑列表
├── 服务中心：服务工单
├── 上机运营：预约管理、上机管理
├── 点餐管理：点餐订单、商品维护
└── 营销活动：团购券、优惠券模板、签到奖励规则、推广计划
```

## 4. 功能需求

### 4.1 认证与账户

| 编号 | 需求 |
|---|---|
| FR-AUTH-001 | 根路径跳转 `/user`；未登录访问用户端时进入 `/user/login`。 |
| FR-AUTH-002 | `/admin` 跳转 `/admin/dashboard`；未登录进入 `/admin/login`。 |
| FR-AUTH-003 | 用户使用身份证号和密码登录，禁用用户不得登录。 |
| FR-AUTH-004 | 注册必填姓名、身份证号、手机号、密码；身份证号和手机号必须唯一。 |
| FR-AUTH-005 | 注册可填写邀请码；有效且活动启用时完成双方奖励。 |
| FR-AUTH-006 | 新用户默认为普通会员、正常状态、余额 0，并生成唯一邀请码。 |
| FR-AUTH-007 | 修改密码必须校验原密码；用户端和管理员端分别退出。 |

### 4.2 会员与余额

| 编号 | 需求 |
|---|---|
| FR-MEMBER-001 | 后台按姓名、身份证号、手机号查询会员。 |
| FR-MEMBER-002 | 后台可新增、编辑、删除会员，并维护余额、等级、类型和状态。 |
| FR-MEMBER-003 | 新增/修改身份证号时同步登录账号；编辑时密码留空保留旧密码。 |
| FR-MEMBER-004 | 后台和用户端充值必须校验金额大于 0，在事务中更新余额并写充值记录。 |
| FR-MEMBER-005 | 散客首次充值后自动转为普通会员。 |
| FR-MEMBER-006 | 用户可查看充值、点餐和上机合并后的余额明细。 |

### 4.3 电脑、上机与预约

| 编号 | 需求 |
|---|---|
| FR-ONLINE-001 | 电脑支持空闲、使用中、预约锁定、维修中四种状态。 |
| FR-ONLINE-002 | 只有正常会员、无进行中记录、目标电脑空闲且余额足够时才能上机。 |
| FR-ONLINE-003 | 开始上机立即按会员折扣扣除首个计费小时。 |
| FR-ONLINE-004 | 后端每 60 秒按当前计费小时补扣新增差额；不足一小时按一小时。 |
| FR-ONLINE-005 | 余额不足以支付新增小时自动结束记录并释放电脑。 |
| FR-ONLINE-006 | 用户和管理员可主动下机；历史记录保留消费与机位快照。 |
| FR-ONLINE-007 | 当前版本不提供换机接口；历史兼容字段不代表页面具备换机功能。 |
| FR-RESERVE-001 | 用户只能预约当前至未来 1 小时内的空闲电脑。 |
| FR-RESERVE-002 | 同一会员同一时间只允许一条有效预约。 |
| FR-RESERVE-003 | 预约成功后电脑变为预约锁定；取消、超时后恢复空闲。 |
| FR-RESERVE-004 | 预约上机必须校验记录归属与有效期，并复用普通上机计费。 |

### 4.4 点餐、优惠券与支付

| 编号 | 需求 |
|---|---|
| FR-FOOD-001 | 后台维护商品名称、分类、价格、上下架状态和备注。 |
| FR-FOOD-002 | 用户端只展示上架商品，购物车可批量提交多个商品。 |
| FR-FOOD-003 | 后端按数据库商品价格计算优惠前金额和实付金额，不信任前端金额。 |
| FR-FOOD-004 | 会员可选可使用且未过期的优惠券，满足门槛后抵扣；重复提交不得重复使用。 |
| FR-PAY-001 | 余额支付立即扣余额并把订单置为“已下单”。 |
| FR-PAY-002 | 微信支付使用 API v3 Native 二维码；支付宝使用电脑网站支付页面。 |
| FR-PAY-003 | 第三方订单先为“待支付”，验签成功的异步通知才转为“已下单”。 |
| FR-PAY-004 | 默认 15 分钟未支付自动关闭，并归还订单占用的优惠券。 |
| FR-PAY-005 | 已下单订单可完成；已完成订单不可取消。 |
| FR-PAY-006 | 余额订单取消时退回该明细实付金额；第三方订单按明细原路退款。 |
| FR-PAY-007 | 支付、退款通知必须验签，重复通知不得重复入账或重复退款。 |
| FR-PAY-008 | 微信、支付宝未启用时返回明确未配置提示，不能影响余额支付。 |

订单业务状态包括：`待支付`、`已下单`、`退款中`、`已完成`、`已取消`。支付主记录状态包括：`待支付`、`已支付`、`已关闭`、`部分退款`、`退款中`、`已退款`。

### 4.5 每日签到与优惠券

| 编号 | 需求 |
|---|---|
| FR-COUPON-001 | 后台维护优惠券名称、使用门槛、优惠金额、有效天数和启停状态。 |
| FR-COUPON-002 | 后台按连续签到天数配置奖励，同一天数只能存在一条规则。 |
| FR-COUPON-003 | 用户同一自然日只能签到一次；断签后连续天数从 1 重新开始。 |
| FR-COUPON-004 | 达到奖励天数时自动发放优惠券，并保存来源、领取时间和过期时间。 |
| FR-COUPON-005 | 优惠券状态为可使用、已使用或已过期；查询时自动标记到期券。 |
| FR-COUPON-006 | 未支付订单关闭或整个批次取消时，应归还已占用优惠券。 |

### 4.6 团购券与推广计划

| 编号 | 需求 |
|---|---|
| FR-VOUCHER-001 | 后台录入唯一券码、充值金额、有效期和备注。 |
| FR-VOUCHER-002 | 用户仅能核销后台存在、未使用、未禁用且未过期的券码。 |
| FR-VOUCHER-003 | 核销采用条件更新防止重复使用，并写入充值记录和券码关联。 |
| FR-VOUCHER-004 | 已使用团购券不可重新启用或禁用。 |
| FR-PROMO-001 | 每个会员拥有唯一邀请码，旧会员升级时自动生成 `WG` 加六位编号格式。 |
| FR-PROMO-002 | 注册填写有效邀请码且活动启用时，邀请人和新用户分别获得配置金额。 |
| FR-PROMO-003 | 一个新用户只能领取一次邀请奖励，奖励记录保存双方和金额快照。 |
| FR-PROMO-004 | 后台可设置双方奖励金额并启用或停用活动；金额不得小于 0。 |

默认推广规则为邀请人 10 元、新用户 5 元，实际运行以数据库 `promotion_rule` 为准。

### 4.7 服务中心

| 编号 | 需求 |

| FR-SERVICE-002 | 故障报修必须选择电脑；呼叫网管可填写所在位置而不绑定电脑。 |
| FR-SERVICE-003 | 同一电脑不能同时存在多条待处理/处理中的故障报修。 |
| FR-SERVICE-004 | 工单状态按待处理 → 处理中 → 已完成流转，也可在未结束前取消。 |
| FR-SERVICE-005 | 故障工单受理时电脑变为维修中；完成或取消处理中的报修时恢复空闲。 |
| FR-SERVICE-006 | 使用中或预约锁定电脑不得直接进入维修中，必须先释放机位。 |

## 5. 核心业务规则

| 编号 | 规则 |
|---|---|
| BR-001 | 金额使用 `BigDecimal`/`DECIMAL`，不使用浮点数直接保存余额。 |
| BR-002 | 上机按计费小时向上取整，首小时立即扣款，之后只补扣差额。 |
| BR-003 | 会员、电脑和业务记录的联动操作使用事务。 |
| BR-004 | 预约和上机必须以后端当前状态为准，前端展示不构成最终授权。 |
| BR-005 | 一次购物车结算共用一个 `batch_no`，第三方支付共用一条支付主记录。 |
| BR-006 | 优惠金额按订单明细依次分摊，取消明细只退该明细的实付金额。 |
| BR-007 | 第三方支付成功、退款成功以验签后的平台回调或主动查询结果为准。 |
| BR-008 | 定时任务每 60 秒执行上机结算、预约过期和支付超时关闭。 |
| BR-009 | 服务工单、团购券、签到、邀请奖励均必须防止重复提交产生重复业务结果。 |
| BR-010 | 所有列表当前均为不分页查询，适合课程演示数据量。 |

## 8. 非功能需求与当前风险

### 8.1 可用性

- 用户端和后台均提供明确中文状态与错误提示；
- 高风险操作使用确认框；
- 表格在窄屏可滚动，后台侧栏可折叠；
- 日期在前端统一格式化，避免直接显示 `T`。

### 8.2 可靠性

- 充值、上机、预约、点餐、团购券、签到和推广奖励采用事务；
- 支付单号、退款单号、邀请码、新用户邀请奖励等有唯一约束；
- 定时任务单条失败应记录日志并继续处理其他记录。

### 8.3 安全边界

当前密码明文保存，后端接口没有统一认证授权，用户编号可从请求路径或参数传入；因此系统不能视为生产级安全系统。上线前必须增加密码散列、后端鉴权、接口权限、CSRF/重放防护、限流、审计和敏感数据脱敏。

支付私钥必须通过服务器文件路径加载，不得进入前端、数据库、Git、聊天或截图。生产回调必须使用 HTTPS。

### 9.3 部署验收

- `http://127.0.0.1:8087/dashboard/statistics` 返回业务成功；
- `https://域名/api/dashboard/statistics` 经 Nginx 返回同样结果；
- `/user`、`/admin` 及深层路由刷新不返回 404；
- 微信/支付宝启用时，支付与退款回调可通过公网 HTTPS 到达后端。
