# 1. 引言

## 1.1 编写目的

本文档用于完整描述“网管通电竞网吧运营管理系统”的建设目标、使用角色、功能范围、业务规则、数据要求、接口要求、非功能要求和验收标准。

本文档的主要读者包括：

- 项目指导教师和答辩评审人员；
- 项目开发、测试和维护人员；
- 需要了解系统功能的网吧管理人员；
- 后续进行二次开发的学习者。

本文档同时作为系统设计、编码、测试和验收的共同依据。需求编号在全文中保持唯一，便于后续定位功能和测试结果。

## 1.2 项目背景

网吧日常运营涉及会员注册与充值、电脑机位维护、用户上机计费、电脑预约、商品点餐和消费记录查询等业务。传统人工处理方式容易出现以下问题：

- 前台在高峰期需要重复处理上机、充值和点餐，工作压力较大；
- 电脑状态依靠人工记录，容易出现状态更新不及时；
- 用户无法实时查看上机时长、消费金额和账户余额；
- 预约未按时上机时，电脑可能被长期占用；
- 点餐订单容易遗漏，订单状态缺少统一管理；
- 用户难以集中查看充值、上机和订单历史。

因此，本项目建设一个前后端分离的网吧运营管理系统，通过管理员后台统一管理基础资料和业务记录，并通过用户自助端降低前台重复操作成本。

## 1.3 项目目标

系统应实现以下目标：

1. 对用户、电脑、商品和业务记录进行统一数字化管理。
2. 实现从选择电脑、开始上机、实时扣费到下机释放的业务闭环。
3. 实现最近一小时内的电脑预约及超时自动取消。
4. 支持商品分类、购物车、自助下单和后台订单处理。
5. 支持用户查询余额明细、上机记录和订单记录。
6. 同时支持本地开发运行和宝塔服务器部署。

## 1.4 系统范围

本期系统包含：

- 用户注册、登录、退出和修改密码；
- 管理员登录和后台导航；
- 会员资料、会员等级、状态和余额管理；
- 电脑机位、区域、价格和状态管理；
- 用户自助上机、管理员办理上机和下机；
- 实时计费、会员折扣、余额提醒和余额不足自动下机；
- 最近一小时内预约、预约锁定、取消和超时释放；
- 模拟充值和充值记录；
- 商品分类、商品上下架、购物车和订单管理；
- 后台经营数据看板和电脑座位图；
- 用户个人消费记录查询；
- 本地运行和宝塔部署支持。

本期系统不包含：

- 微信、支付宝、银行卡等真实支付接口；
- 短信验证码和实名认证接口；
- 电脑客户端远程开机、关机或锁屏控制；
- 商品库存、采购和供应商管理；
- 复杂财务报表、发票和税务处理；
- 多门店、加盟店和跨门店会员体系；
- 生产级身份认证、权限模型和安全审计。

# 2. 总体描述

## 2.1 产品定位

本系统定位为适合课程演示和中小型网吧业务原型验证的运营管理系统。系统以浏览器为主要运行载体，用户无需安装独立客户端即可使用。

系统由四部分组成：

```text
用户自助端 / 管理员后台
          ↓ HTTP 请求
       Spring Boot 后端
          ↓ MyBatis-Plus
         MySQL 数据库
```

## 2.2 用户角色

| 角色 | 说明 | 主要能力 |
|---|---|---|
| 未登录访客 | 尚未登录系统的访问者 | 进入用户登录/注册页或管理员登录页 |
| 散客用户 | 数据库中用户类型为“散客”的正常用户 | 登录、查看资料、点餐、模拟充值；充值后可转为普通会员 |
| 普通会员 | 基础正式会员 | 自助上机、预约、点餐、充值、查看记录；上机不打折 |
| 黄金会员 | 高级会员 | 普通会员全部能力；上机 9 折 |
| 钻石会员 | 高级会员 | 普通会员全部能力；上机 8 折 |
| 管理员 | 网吧工作人员 | 管理用户、电脑、充值、上机、预约、商品、订单和经营数据 |

## 2.3 运行环境

### 2.3.1 客户端环境

- 支持现代桌面浏览器，如 Chrome、Edge 和 Safari；
- 用户端以电脑大屏和网吧自助终端为主要适配目标；
- 窄屏设备应能纵向滚动并保持主要功能可操作；
- 浏览器需要启用 JavaScript 和 LocalStorage。

### 2.3.2 服务端环境

- JDK 17；
- Maven 3.8 或以上；
- Node.js 18 或以上；
- MySQL 8；
- Spring Boot 3；
- Linux 宝塔或本地 macOS、Windows、Linux 开发环境。

## 2.4 技术约束

- 前端使用 Vue 3、Vue Router、Axios、Vite 和 Arco Design Vue；
- 后端使用 Spring Boot、MyBatis-Plus 和 MySQL；
- 后端默认端口为 `8080`；
- 前端本地开发默认端口为 `5173`；
- 日期时间接口统一使用 `yyyy-MM-dd HH:mm:ss` 或可被前端格式化的 ISO 时间；
- 金额使用 `BigDecimal` 和数据库 `DECIMAL`，不得使用浮点数直接保存余额；
- 前后端接口统一返回 `code`、`message` 和 `data`。

## 2.5 假设与依赖

- MySQL 服务已正确启动并完成数据库初始化；
- 后端数据库账号具有读写 `wangguantong` 数据库的权限；
- 用户和管理员能够通过浏览器访问前端与后端服务；
- 定时任务功能在后端启动类中保持启用；
- 网吧工作人员负责维护真实的电脑区域、价格和维修状态；
- 模拟充值仅用于课堂展示，不代表真实资金已经到账。

# 4. 功能需求

## 4.1 访问入口与登录控制

| 编号 | 需求名称 | 需求说明 | 优先级 |
|---|---|---|---|
| FR-AUTH-001 | 默认用户入口 | 访问根路径 `/` 时，系统应跳转至用户系统；未登录时进入 `/user/login` | 高 |
| FR-AUTH-002 | 管理员入口 | 管理员通过 `/admin` 进入后台；未登录时跳转至 `/admin/login` | 高 |
| FR-AUTH-003 | 用户端布局隔离 | 用户端和用户登录页不得显示管理员侧边栏和后台头部 | 高 |
| FR-AUTH-004 | 后台布局隔离 | 后台业务页面应显示统一侧边菜单、顶部管理员信息和退出按钮 | 高 |
| FR-AUTH-005 | 独立登录状态 | 用户和管理员应分别保存登录状态，互不替代 | 高 |
| FR-AUTH-006 | 退出登录 | 退出后系统应清除对应登录状态并返回相应登录页 | 高 |

## 4.2 用户注册、登录与密码管理

| 编号 | 需求名称 | 需求说明 | 优先级 |
|---|---|---|---|
| FR-USER-001 | 用户注册 | 用户应能填写姓名、身份证号、手机号和密码注册账号 | 高 |
| FR-USER-002 | 注册必填校验 | 姓名、身份证号、手机号和密码任意一项为空时不得注册 | 高 |
| FR-USER-003 | 身份证号唯一 | 用户注册时，身份证号已存在应提示“该身份证号已注册” | 高 |
| FR-USER-004 | 手机号唯一 | 用户注册时，手机号已存在应提示“该手机号已注册” | 高 |
| FR-USER-005 | 注册默认属性 | 注册成功后，登录账号应等于身份证号，余额为 0，用户类型为会员，级别为普通会员，状态为正常 | 高 |
| FR-USER-006 | 用户登录 | 用户应使用身份证号和密码登录 | 高 |
| FR-USER-007 | 登录状态校验 | 账号状态不是“正常”时，应拒绝登录并提示联系前台 | 高 |
| FR-USER-008 | 修改密码 | 已登录用户应能输入原密码、新密码和确认密码修改密码 | 高 |
| FR-USER-009 | 原密码校验 | 原密码错误时不得修改密码 | 高 |
| FR-USER-010 | 两次密码一致 | 两次输入的新密码不一致时，前端不得提交 | 高 |

| FR-ONLINE-003 | 单用户单机 | 一个会员同一时间只允许存在一条“进行中”上机记录 | 高 |
| FR-ONLINE-004 | 电脑空闲校验 | 只有状态为“空闲”的电脑可以开始上机 | 高 |
| FR-ONLINE-005 | 首小时扣费 | 上机成功时应立即扣除折扣后的首小时费用 | 高 |
| FR-ONLINE-006 | 首小时余额校验 | 余额不足以支付首小时费用时不得上机 | 高 |

# 5. 核心用例

## 5.1 UC-01 用户自助上机

| 项目 | 内容 |
|---|---|
| 用例名称 | 用户自助上机 |
| 参与者 | 普通会员、黄金会员、钻石会员 |
| 前置条件 | 用户已登录、状态正常、当前没有进行中记录、目标电脑为空闲、余额足以支付首小时 |
| 触发条件 | 用户在空闲电脑卡片上点击“上机”并确认 |
| 后置条件 | 会员余额扣除首小时费用，新增进行中记录，电脑变为使用中 |

主流程：

1. 系统展示按区域分组的电脑座位图。
2. 用户选择一台空闲电脑并点击“上机”。
3. 系统显示“确认后立即开始计费”的确认信息。
4. 用户确认上机。
5. 后端验证用户状态、重复上机、电脑状态和余额。
6. 后端计算折扣后的首小时费用并扣款。
7. 后端新增进行中上机记录并占用电脑。
8. 页面切换为当前上机电脑视图。

异常流程：

- 用户状态异常：拒绝上机并提示联系前台；
- 已有进行中记录：拒绝重复上机；
- 电脑已被使用、预约或维修：拒绝上机；
- 余额不足：拒绝上机并提示充值；
- 请求期间数据状态变化：以后端最终校验结果为准。

## 5.2 UC-02 实时扣费与自动下机

| 项目 | 内容 |
|---|---|
| 用例名称 | 实时扣费与余额不足自动下机 |
| 参与者 | 正在上机的会员、系统定时任务 |
| 前置条件 | 存在状态为“进行中”的上机记录 |
| 触发条件 | 定时任务执行、用户查询记录或用户点击下机 |
| 后置条件 | 余额充足时补扣新增费用；余额不足时结束记录并释放电脑 |

主流程：

1. 系统计算从开始时间到当前时间的分钟数。
2. 系统将分钟数向上取整为计费小时，最低为 1 小时。
3. 系统按小时单价、计费小时和会员折扣计算累计应收金额。
4. 累计应收金额减去记录中的已扣金额，得到本次待扣差额。
5. 余额充足时扣除差额，并更新记录累计金额。
6. 余额不足时结束记录、记录结束时间并释放电脑。

## 5.3 UC-03 用户预约电脑

| 项目 | 内容 |
|---|---|
| 用例名称 | 用户预约电脑 |
| 参与者 | 正常会员 |
| 前置条件 | 用户已登录、目标电脑为空闲 |
| 触发条件 | 用户点击“预约”，选择时间后确认 |
| 后置条件 | 新增已预约记录，电脑变为预约锁定 |

主流程：

1. 用户选择一台空闲电脑。
2. 用户选择当前时间至未来一小时之间的预约时间。
3. 系统验证用户、电脑和时间。
4. 系统保存预约记录。
5. 系统将电脑改为预约锁定。

超时流程：预约时间到达后仍未上机，系统自动取消记录并释放电脑。

## 5.4 UC-04 用户购物车下单

| 项目 | 内容 |
|---|---|
| 用例名称 | 用户自助点餐 |
| 参与者 | 已登录且状态正常的用户 |
| 前置条件 | 存在已上架商品，用户余额足够 |
| 触发条件 | 用户选择商品数量后点击“下单” |
| 后置条件 | 创建订单、扣除余额、清空购物车并刷新订单记录 |

主流程：

1. 系统按分类展示上架商品。
2. 用户使用商品卡片上的加减号选择数量。
3. 系统在购物车中显示清单和总金额。
4. 用户点击“下单”。
5. 系统校验购物车、商品状态和余额。
6. 后端根据数据库商品单价计算订单金额。
7. 系统扣除余额并创建订单。
8. 页面清空购物车并刷新余额与订单记录。

## 5.5 UC-05 模拟充值

| 项目 | 内容 |
|---|---|
| 用例名称 | 用户模拟充值 |
| 参与者 | 已登录用户 |
| 前置条件 | 用户账号存在 |
| 触发条件 | 用户选择或输入金额并确认充值 |
| 后置条件 | 余额增加，生成充值记录；散客转为普通会员 |

异常流程：金额小于等于 0、超过 10000 元或重复提交时，不得发起有效充值。

# 6. 业务规则

| 编号 | 规则名称 | 规则说明 |
|---|---|---|
| BR-001 | 身份证号登录 | 用户登录账号使用身份证号 |
| BR-002 | 正常状态 | 禁用或异常用户不得登录和执行用户端核心操作 |
| BR-003 | 普通会员折扣 | 普通会员折扣系数为 1.00，即不打折 |
| BR-004 | 黄金会员折扣 | 黄金会员折扣系数为 0.90 |
| BR-005 | 钻石会员折扣 | 钻石会员折扣系数为 0.80 |
| BR-006 | 计费小时 | 上机时长不足一小时按一小时计算，超过整点部分向上取整 |
| BR-007 | 实时扣费 | 上机时立即扣首小时，进入新计费小时后只补扣新增差额 |
| BR-008 | 禁止欠费 | 余额不足以支付新增计费小时，不允许继续产生欠费，应自动下机 |
| BR-009 | 单用户单机 | 同一会员不得同时存在多条进行中上机记录 |
| BR-010 | 空闲可操作 | 只有空闲电脑允许普通上机或创建预约 |
| BR-011 | 预约范围 | 预约时间必须位于当前时间至未来一小时之间 |
| BR-012 | 预约超时 | 超过预约时间仍未上机，预约自动取消 |
| BR-013 | 散客转会员 | 散客充值后自动成为普通会员 |
| BR-014 | 商品价格来源 | 订单单价必须从数据库商品资料读取 |
| BR-015 | 会员订单付款 | 会员订单在创建时立即从余额扣款 |
| BR-016 | 订单取消退款 | 未完成会员订单取消后，原金额退回余额 |
| BR-017 | 已完成不可取消 | 已完成订单不得取消或退款 |
| BR-018 | 金额精度 | 所有金额保留两位小数并采用四舍五入规则 |
| BR-019 | 业务快照 | 历史记录保存当时的会员姓名、电脑编号、商品名称和价格 |
| BR-020 | 模拟支付 | 用户充值不代表真实支付，仅修改系统内余额 |

计费公式：

```text
计费小时 = max(1, ceil(上机分钟数 / 60))
累计费用 = 电脑小时单价 × 计费小时 × 会员折扣系数
待扣金额 = 累计费用 - 上机记录已扣金额
```

- 请求超时时间默认为 10 秒；
- 网络失败时，前端应提示“服务器连接失败”；
- 业务失败时，前端应显示后端返回的 `message`。

# 9. 用户界面需求

## 9.1 总体视觉要求

- 用户端采用浅色、简洁、商务风格，不使用大面积深色背景；
- 管理员后台保持安静、清晰、适合高频操作的管理系统风格；
- 卡片圆角不宜过大，避免过度装饰；
- 状态颜色保持一致，不同页面不得对同一状态使用相反颜色；
- 页面文字不得被按钮、标签或表格列遮挡；
- 所有操作按钮文字不得换行；
- 日期时间统一显示为可读格式，不得显示字母 `T`；
- 输入框应有清晰边框、悬停状态和聚焦状态。

## 9.2 用户端布局要求

- 桌面端应提供明确的功能导航、账户摘要和业务工作区；
- 座位图应优先展示机位编号、状态、区域和价格；
- 上机后只展示当前电脑，不得继续展示其他机位供用户操作；
- 点餐商品列表应使用紧凑高度，避免单个商品卡片占用过多纵向空间；
- 购物车宽度应随页面适配，不得只占左侧小区域或留下异常空白；
- 购物车应显示完整清单，但数量只通过商品卡片的加减号调整；
- 余额不足、预约限制和自动下机等提示必须包含可读文案，不得只显示图标。

## 9.3 响应式要求

- 宽屏桌面端应充分利用横向空间；
- 平板宽度下，侧边导航可转换为顶部导航；
- 内容和右侧购物车在空间不足时应改为单列；
- 手机宽度下，机位卡片、商品卡片和实时指标应能单列显示；
- 页面允许纵向滚动，但不得出现无意义的横向页面滚动。

# 10. 非功能需求

## 10.1 性能需求

| 编号 | 要求 |
|---|---|
| NFR-PERF-001 | 在正常本地网络和课堂数据量下，普通列表查询目标响应时间不超过 3 秒 |
| NFR-PERF-002 | 页面按钮提交后应提供成功或失败反馈，不得无响应 |
| NFR-PERF-003 | 前端上机时长每秒在本地刷新，不应每秒请求后端 |
| NFR-PERF-004 | 上机期间前端约每 30 秒同步一次后端业务数据 |
| NFR-PERF-005 | 后端计费和预约定时任务执行间隔约为 60 秒 |

## 10.2 可用性需求

| 编号 | 要求 |
|---|---|
| NFR-USE-001 | 表单必填项缺失时应给出明确中文提示 |
| NFR-USE-002 | 删除、上机、下机、取消预约和订单状态操作应提供确认机制 |
| NFR-USE-003 | 查询表单应提供重置功能 |
| NFR-USE-004 | 操作成功后应刷新相关列表、余额和状态 |
| NFR-USE-005 | 没有数据时应显示空状态，而不是空白页面或报错 |

## 10.3 可靠性与一致性需求

| 编号 | 要求 |
|---|---|
| NFR-REL-001 | 涉及余额和业务记录的多表修改应使用事务 |
| NFR-REL-002 | 单条定时计费失败不得阻止其他上机记录继续结算 |
| NFR-REL-003 | 下机、取消预约后必须释放对应电脑 |
| NFR-REL-004 | 页面显示的余额和消费应以后端数据为最终依据 |
| NFR-REL-005 | 金额计算统一保留两位小数 |

## 10.4 安全需求与当前边界

课堂版当前要求：

- 用户和管理员页面使用独立登录入口；
- 未登录访问受保护页面时进行前端路由跳转；
- 密码输入框使用遮罩显示；
- 用户状态异常时拒绝用户端登录；
- 前端不直接决定商品价格、折扣和最终扣款金额。

正式生产版本必须补充：

- 使用 BCrypt 等算法加密存储密码；
- 使用 Spring Security 配合 Session 或 JWT 完成后端认证；
- 后端根据登录身份判断数据归属，禁止仅凭前端提交的 `memberId` 查询他人记录；
- 增加接口权限、登录失效、频率限制和操作审计；
- 对身份证号和手机号进行脱敏展示；
- 对真实支付接口进行签名和回调验证；
- 使用数据库约束或锁处理并发抢机和重复预约。

## 10.5 可维护性需求

| 编号 | 要求 |
|---|---|
| NFR-MAIN-001 | 后端保持 Controller、Service、Mapper、Entity 分层结构 |
| NFR-MAIN-002 | 前端接口函数集中放在 `src/api`，页面不得重复硬编码完整接口地址 |
| NFR-MAIN-003 | 时间格式化、请求错误处理等公共逻辑应集中封装 |
| NFR-MAIN-004 | 状态值和折扣规则修改时，应同步检查前端展示、后端逻辑和数据库脚本 |
| NFR-MAIN-005 | 部署参数通过配置文件或环境变量维护，不写死生产服务器地址 |

## 10.6 部署需求

- 后端应能通过 Maven 构建为可执行 JAR；
- 前端应能通过 `npm run build` 生成静态资源；
- 本地环境应能分别启动前端和后端；
- 宝塔环境应能通过 Java 项目或命令运行 JAR；
- Nginx 应能托管前端静态文件，并将 `/api` 请求代理至后端；
- 部署文档应包含数据库导入、配置修改、端口和常见错误排查。

# 11. 验收标准

## 11.1 功能验收

| 编号 | 验收场景 | 预期结果 |
|---|---|---|
| AC-001 | 未登录访问 `/user` | 自动进入用户登录页 |
| AC-002 | 使用正确身份证号和密码登录 | 进入用户端并显示本人资料与余额 |
| AC-003 | 使用重复身份证号或手机号注册 | 注册失败并显示对应提示 |
| AC-004 | 未登录访问 `/admin/dashboard` | 自动进入管理员登录页 |
| AC-005 | 管理员新增会员但未填密码 | 保存失败并提示设置密码 |
| AC-006 | 管理员新增或修改电脑 | 电脑列表和座位图显示最新数据 |
| AC-007 | 正常会员对空闲电脑确认上机 | 首小时余额立即扣除，电脑变为使用中 |
| AC-008 | 余额不足以支付首小时 | 上机失败，余额、电脑和记录均不发生变化 |
| AC-009 | 用户已在上机 | 座位页只显示当前电脑和下机功能 |
| AC-010 | 余额不足以支付新增小时 | 系统自动下机，记录完成，电脑释放 |
| AC-011 | 用户自助下机 | 记录完成并显示结束时间，电脑恢复空闲 |
| AC-012 | 选择超过未来一小时的预约时间 | 预约失败并提示时间范围 |
| AC-013 | 有效预约成功 | 生成已预约记录，电脑变为预约锁定 |
| AC-014 | 预约超时未上机 | 预约自动取消，电脑恢复空闲 |
| AC-015 | 普通/黄金/钻石会员分别上机 | 分别按不打折、9 折、8 折计算 |
| AC-016 | 散客成功充值 | 余额增加，并转为普通会员 |
| AC-017 | 商品数量从 0 增加 | 商品进入购物车，清单和总额正确 |
| AC-018 | 购物车下单且余额充足 | 创建订单、扣除余额并清空购物车 |
| AC-019 | 购物车总额超过余额 | 不创建订单，并提示余额不足 |
| AC-020 | 后台完成已下单订单 | 状态变为已完成，不再显示取消操作 |
| AC-021 | 后台取消会员订单 | 状态变为已取消，金额退回余额 |
| AC-022 | 用户查看消费记录 | 能切换余额、上机和订单记录，且只显示本人数据 |
| AC-023 | 页面显示后端时间 | 时间格式中不出现字母 `T` |
| AC-024 | 操作列空间较窄 | 按钮保持同一行，文字不换行或遮挡 |
| AC-025 | 用户确认模拟充值 | 余额和充值记录增加，页面提示未接真实支付 |

## 11.2 构建验收

- 后端执行 `mvn -DskipTests package` 应成功生成 JAR；
- 前端执行 `npm install` 后，`npm run build` 应成功生成 `dist`；
- 前后端启动后，用户端和管理员后台均可访问；
- 浏览器控制台不得出现导致核心功能不可用的错误；
- 数据库连接失败时，后端应有可定位的错误日志。

## 11.3 数据验收

- 上机、预约、订单和充值操作完成后，对应业务表应生成或更新正确记录；
- 余额变动应与充值、点餐和上机记录一致；
- 电脑状态应与进行中的上机记录或有效预约保持一致；
- 取消订单不得造成重复退款；
- 已完成上机记录不得继续扣费。

# 12. 需求优先级与版本边界

## 12.1 优先级定义

| 高 | 核心业务或数据正确性要求，缺失时系统无法完成主要流程 |
| 中 | 影响操作效率、展示质量或管理完整性，但不阻断核心流程 |
| 低 | 可在后续版本优化的辅助能力 |

## 12.2 当前版本已覆盖能力

- 用户端和管理员后台双入口；
- 用户注册、身份证号登录和修改密码；
- 会员、电脑、商品基础资料管理；
- 实时上机计费、折扣和余额不足自动下机；
- 最近一小时预约和超时自动取消；
- 模拟充值和散客转普通会员；
- 分类点餐、购物车、订单完成和取消退款；

# 附录 B：课堂版风险说明

本说明书以当前项目已经具备的业务功能为主体。由于项目定位为课堂作业版本，登录状态主要由前端 LocalStorage 保存，密码当前未进行生产级加密，部分用户记录接口以路径中的会员编号查询数据，充值也没有连接真实支付平台。

因此，本项目适合课程演示、功能学习和原型验证，不应未经安全改造直接投入真实商业运营。若用于正式生产，应优先落实第 10.4 节中的安全要求，并完成并发、压力、数据备份和故障恢复测试。
